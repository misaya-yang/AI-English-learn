## 第一优先级：基础设施加固（P0）

### 1.1 数据层从 localStorage 迁移到 IndexedDB-First

**现状问题：** localStorage 有 5-10MB 限制，同步的 JSON.parse/stringify 阻塞 UI 线程，超过 1000 词汇后性能急剧下降。

**设计方案：**

- 将所有学习进度数据（word_progress、review_logs、sessions）完全迁移到 IndexedDB
- 使用 `idb` 库的异步 API 替换所有同步 localStorage 调用
- 建立复合索引 `[userId, status]`、`[userId, nextReview]` 以支持高效查询
- 实现 LRU 缓存策略：内存中保持热数据（当日词汇 + 待复习），冷数据按需加载
- 添加数据压缩（对 word definitions 使用 LZ-String 压缩存储）

**预计收益：** 支撑 10,000+ 词汇量用户；UI 线程不再阻塞；存储空间利用率提升 60%。

### 1.2 离线同步引擎重构

**现状问题：** 无冲突解决策略；逐条同步（无批量）；指数退避起点过快（1s）；review_logs 无限增长。

**设计方案：**

- **冲突解决：** 采用 Last-Write-Wins + Vector Clock 混合策略
  - 对 FSRS 状态使用 LWW（最后一次复习结果覆盖）
  - 对 learning_events 使用 append-only（永不冲突，服务端合并）
  - 对 user_profile 使用 field-level merge（各字段独立合并）
- **批量同步：** 将 sync_queue 中的 ops 按表分组，使用 Supabase batch upsert
- **退避策略：** 起始 3s，最大 60s，加入 jitter 避免惊群
- **数据清理：** review_logs 保留 180 天，超期数据聚合为日级摘要后删除明细
- **乐观更新：** UI 层先更新本地状态，同步失败时回滚 + toast 提示

**预计收益：** 离线学习体验与在线一致；同步带宽降低 70%；数据存储可控增长。

### 1.3 统一数据源（消除双重存储）

**现状问题：** custom_words 同时存在于 CUSTOM_WORDS 和 word_books 两个存储中；进度数据在 localStorage 和 IndexedDB 之间重复。

**设计方案：**

- 以 IndexedDB 为单一数据源（Single Source of Truth）
- localStorage 仅保留轻量级配置（theme、language_preference）
- custom_words 统一归入 word_books 体系，通过 `source: 'user'` 标识
- 建立迁移脚本，一次性将 localStorage 数据迁移至 IndexedDB

## 第二优先级：学习智能升级（P1）

### 2.1 FSRS 个性化参数调优

**现状问题：** 17 个 FSRS 权重参数全局固定，不区分用户水平和记忆特征。

**设计方案：**

- **用户级参数拟合：** 当用户积累 200+ 条复习记录后，使用梯度下降在客户端拟合个人化参数
  - 在 Web Worker 中运行，不阻塞 UI
  - 每周重新拟合一次，存储到 user_learning_profiles
- **冷启动策略：** 新用户根据 CEFR 等级选择预训练参数集（A1-A2、B1-B2、C1-C2 各一组）
- **时间感知调度：** 记录每次复习的时间点，发现用户习惯学习时段，将复习推送到记忆效果最佳的时间窗口
- **Lapse 计数利用：** 当某词 lapse >= 3 次，自动标记为"顽固词"，切换到强化学习路径（缩短间隔 + 多模态复习）

**预计收益：** 记忆保持率从 90% 目标提升到实际 92-95%；顽固词突破率提升。

### 2.2 学习者模型进化：从规则到预测

**现状问题：** 学习模式（recovery/steady/stretch/sprint）基于硬阈值切换；燃尽风险是简单启发式；弱点检测仅依赖事件计数。

**设计方案：**

- **预测模型：**
  - 使用轻量 logistic regression 预测"明天是否会学习"（基于最近 14 天活跃模式 + 当日待复习量）
  - 预测"本次复习是否会记住"（基于 FSRS 状态 + 上次学习间隔 + 词汇难度）
  - 结果反馈到每日目标调整：预测不会学习的日子，降低任务量 + 推送提醒
- **燃尽检测升级：**
  - 跟踪连续 3 天的"完成率 < 60%"作为燃尽信号
  - 监控平均会话时长的下降趋势
  - 触发时自动切换到 recovery 模式并减少新词量
- **弱点语义聚类：**
  - 将 error_trace 中的错误按语义聚类（不仅按 tag 计数）
  - 例如：区分"介词搭配错误"和"时态错误"，而不是统归为"grammar"
  - 使用聚类结果驱动练习内容生成

### 2.3 自适应难度系统重构

**现状问题：** 仅有 recover/steady/stretch 三档，practice 难度实际未调整（仅 UI 标签变化）。

**设计方案：**

- **连续难度评分（0-100）：**
  - 综合因素：当前 retrievability 分布、近 7 日正确率、streak 长度、待复习积压量
  - 公式：`difficulty = w1 × (1 - avgR) + w2 × (1 - accuracy7d) + w3 × backlogRatio - w4 × streakBonus`
  - 每次学习会话前重新计算
- **练习内容难度匹配：**
  - 为每个 quiz 题目标注难度分（基于词汇 CEFR 等级 + 干扰项相似度）
  - 使用 difficulty score 选择与用户水平匹配的题目
  - 实现 zone of proximal development（最近发展区）策略：80% 题目在舒适区，20% 略高于当前水平
- **动态任务量调整：**
  - 根据实际用时/预估用时的比值调整后续任务量
  - 如果用户连续完成 3 次 "stretch" 难度且正确率 > 85%，自动升级基线难度

## 第三优先级：AI 能力深化（P1）

### 3.1 多模型路由架构

**现状问题：** 所有 AI 请求走 DeepSeek 单一模型，无法针对不同场景优化成本和质量。

**设计方案：**

- **场景分层路由：**
  - 简单问答/翻译 → 轻量模型（DeepSeek-Lite 或 Qwen-Turbo）：低延迟、低成本
  - 深度解释/作文批改 → 主力模型（DeepSeek-Chat）：高质量
  - 内容生成/考试模拟 → 高能力模型（Claude/GPT-4 级别）：最高质量
- **路由决策器：**
  - 基于输入长度、任务类型（chat_mode）、用户等级自动选择
  - 支持用户手动切换（"快速回答" vs "深度讲解"）
- **成本控制：**
  - 每用户每日 token 预算
  - 免费用户使用轻量模型，付费用户解锁高能力模型

### 3.2 记忆系统升级

**现状问题：** 嵌入降级为本地 hash（384 维）质量差；记忆去重用校验和（无语义去重）；无层级结构。

**设计方案：**

- **嵌入质量保障：**
  - 接入专用嵌入 API（text-embedding-3-small 或 BGE-M3）
  - 本地缓存嵌入结果到 IndexedDB，避免重复调用
  - 降级时使用 TF-IDF + 余弦相似度替代 hash（比 hash 质量高 3-5 倍）
- **语义去重：**
  - 新记忆入库时与现有记忆计算余弦相似度
  - 相似度 > 0.85 时合并（取置信度较高者 + 更新时间戳）
  - 定期（每周）运行全量去重任务
- **层级记忆：**
  - L1（工作记忆）：最近 3 次对话的上下文，TTL 24 小时
  - L2（短期记忆）：学习事件、错误模式，TTL 45 天
  - L3（长期记忆）：用户画像、学习偏好、已掌握知识体系，永久

### 3.3 对话-学习闭环

**现状问题：** AI 对话中发现的弱点不会反馈到学习者模型；学习引擎不感知对话历史。

**设计方案：**

- **对话事件提取：**
  - AI 对话结束后，后台分析对话内容
  - 自动提取：用户问了什么词/语法点、哪些回答表明理解有误、哪些话题引起了兴趣
  - 将提取结果写入 learning_events（type: 'chat_insight'）
- **学习引擎消费对话洞察：**
  - learnerModel 将 chat_insight 纳入弱点检测
  - 如果对话中反复问同一个语法点，自动生成针对性 grammar lesson
  - 对话中提及的新词自动加入"推荐学习"列表
- **AI 感知学习进度：**
  - 在 context-engine 中注入"本周学习摘要"（新学了哪些词、哪些主题进步了）
  - AI 可以主动说："你这周学了 'resilient' 和 'tenacious'，来试试在句子中用用看？"

## 第四优先级：练习系统重构（P1）

### 4.1 智能干扰项生成

**现状问题：** 干扰项从其他词库词随机选取，无语义关联，辨别难度低。

**设计方案：**

- **语义相关干扰项：**
  - 预计算词汇的语义相似度矩阵（使用 word embeddings）
  - 干扰项从语义最接近的词中选取（如 "abandon" 的干扰项用 "desert"、"forsake"、"neglect"）
  - 根据用户水平调整干扰项难度：初级用户用远义词，高级用户用近义词
- **常见错误驱动：**
  - 收集所有用户的错误选择数据
  - 将最常被误选的选项作为干扰项（data-driven distractors）
  - 例如：如果 30% 的用户在 "affect" 题中选了 "effect"，则始终保留 "effect" 作为干扰项

### 4.2 多元题型扩展

**现状问题：** 仅有选择题和填空题两种模式，题目固定 10 道。

**新增题型设计：**

- **句子排序：** 打乱句子中的词序，要求重新排列（训练语感和语法结构）
- **词根推断：** 给出未知词和词根/词缀提示，推断含义（训练词汇推理能力）
- **语境选词：** 一段文章中若干空白，从给定词汇中选择最合适的（训练上下文理解）
- **错误识别：** 给出一个句子，标出语法/用词错误的位置（训练语法敏感度）
- **听写复述：** 播放一段话，用户打字复述关键内容（训练听力 + 写作 + 记忆）
- **自适应题量：** 根据正确率动态调整：全对则提前结束（省时间），连错则追加巩固题

### 4.3 练习序列智能编排

**设计方案：**

- **间隔交叉练习（Interleaving）：**
  - 不按主题分组出题，而是交叉不同主题的题目
  - 研究表明 interleaving 比 blocking 的长期记忆效果好 20-40%
- **错题重现机制：**
  - 每次练习中混入 20% 的历史错题
  - 错题间隔递增重现（1 天 → 3 天 → 7 天 → 14 天）
- **难度递进策略：**
  - 每组练习从简单题开始（建立信心），逐步提升难度
  - 如果连续答对 3 题，跳级提升难度
  - 如果连续答错 2 题，降低难度并切入解释模式

## 第五优先级：内容与个性化（P2）

### 5.1 学习路径智能推荐

**设计方案：**

- **基于目标的路径生成：**
  - 用户设定目标（通过 IELTS、日常交流、商务英语）
  - 系统反向推算所需词汇量、语法点、技能分布
  - 生成个性化学习路径，标注里程碑
- **动态路径调整：**
  - 每周评估路径进度，如果某个 track 进度落后，自动调整权重
  - 如果某个领域已超前完成，释放时间给薄弱领域
- **主题偏好学习：**
  - 追踪用户在不同主题上的学习时长和参与度
  - 在保证核心目标的前提下，优先推送高兴趣主题的内容

### 5.2 AI 内容生成质量闭环

**现状问题：** AI 生成内容有 validation pipeline，但缺少用户反馈驱动的质量改进。

**设计方案：**

- **用户隐式反馈收集：**
  - 追踪生成内容的使用率（阅读时长、完成率、跳过率）
  - 追踪练习题的区分度（高分和低分用户的正确率差异）
  - 区分度低的题目自动降级或重新生成
- **内容质量评分：**
  - 综合评分 = 0.4 × 完成率 + 0.3 × 区分度 + 0.2 × 用户评价 + 0.1 × AI 审核分
  - 低于阈值的内容自动进入人工审核队列
- **生成提示词迭代：**
  - 记录每个 prompt template 生成内容的平均质量分
  - A/B 测试不同 prompt 变体，自动选择表现最优的版本

### 5.3 社交学习功能

**设计方案：**

- **学习小组：**
  - 支持创建 3-5 人学习小组
  - 共享每日进度（类似 GitHub contribution graph）
  - 组内互相出题/纠错
- **词汇对战：**
  - 实时 1v1 词汇对战（websocket 连接）
  - 限时抢答模式：同时看到题目，先答对者得分
  - 使用双方共同学习过的词汇出题
- **社区词书分享：**
  - 用户可以发布自己整理的词书
  - 其他用户可以 fork + 收藏
  - 基于使用量排名的热门词书榜

## 第六优先级：性能与体验优化（P2）

### 6.1 前端性能优化

**设计方案：**

- **代码分割：**
  - 按路由懒加载（React.lazy + Suspense）
  - 将 Recharts、Framer Motion 等大型库按需加载
  - 预估可以将首屏加载体积减少 40-50%
- **渲染优化：**
  - AnalyticsPage 的 topic breakdown 计算加 useMemo
  - 列表虚拟化：VocabularyBank 使用 react-window 处理大词汇量
  - 减少不必要的 re-render：对 Context 拆分（Auth / Theme / UserData 拆为更细粒度）
- **缓存策略：**
  - Service Worker 缓存静态资源
  - React Query 的 staleTime 针对不同数据设置不同策略
  - 词汇数据使用 indefinite cache（变更通过 invalidation 驱动）

### 6.2 移动端体验增强

**设计方案：**

- **手势交互：**
  - 词汇卡片左滑（忘记）/ 右滑（记住）的 Tinder 式交互
  - 下拉刷新今日任务
  - 长按单词显示详情浮窗
- **离线优先体验：**
  - 提前缓存未来 3 天的待复习词汇
  - 离线状态下所有核心学习功能可用（词汇学习、复习、练习）
  - 恢复在线时静默同步
- **PWA 增强：**
  - 添加 manifest.json 支持安装到主屏幕
  - 推送通知（复习提醒、streak 断裂预警）
  - 后台同步（Background Sync API）

### 6.3 分析仪表盘升级

**设计方案：**

- **预测性分析：**
  - "如果按当前速度学习，预计 X 天后达到 Y 词汇量"
  - "本周遗忘风险最高的 10 个词"（基于 FSRS retrievability 预测）
  - "推荐复习时间窗口"（基于历史学习时间模式）
- **学习洞察：**
  - 自动检测学习模式变化（"你最近 3 天的学习时长下降了 40%"）
  - 对比分析（"你的 B2 词汇掌握速度比平均快 25%"，需匿名化）
  - 弱点趋势图（某个语法点的正确率随时间的变化）
- **可导出报告：**
  - 周报/月报 PDF 导出
  - 包含关键指标、进步亮点、建议改进方向

## 技术决策要点

### 需要做出的关键决策

1. **嵌入模型选择：** text-embedding-3-small（OpenAI，0.02$/1M tokens）vs BGE-M3（自托管，免费但需 GPU）vs Cohere Embed v3（多语言优势）。推荐 BGE-M3（中英双语场景最优）。

2. **FSRS 参数拟合位置：** 客户端（Web Worker）vs 服务端（Edge Function）。推荐客户端 — 保护隐私，减少服务端负载，200+ 条记录的拟合在现代浏览器 < 2s。

3. **实时对战方案：** Supabase Realtime（已有基础设施）vs 独立 WebSocket 服务。推荐先用 Supabase Realtime 快速验证，规模化后再考虑独立服务。

4. **内容生成缓存：** 按需生成 vs 预生成池。推荐混合策略 — 基础题库预生成，个性化内容按需生成。

## 风险评估

| 风险 | 概率 | 影响 | 缓解措施 |
|------|------|------|----------|
| IndexedDB 迁移导致数据丢失 | 中 | 高 | 双写策略：迁移期间同时写 localStorage 和 IDB |
| FSRS 参数拟合不稳定 | 低 | 中 | 设置参数变化范围上限，异常时回退全局参数 |
| AI 成本随用户增长失控 | 高 | 高 | 分层模型路由 + 用户级 token 预算 + 缓存高频问答 |
| 社交功能滥用 | 中 | 中 | 内容审核 + 举报机制 + 速率限制 |
| 移动端性能瓶颈 | 中 | 中 | 虚拟化列表 + 图片懒加载 + 计算密集任务 offload 到 Worker |
